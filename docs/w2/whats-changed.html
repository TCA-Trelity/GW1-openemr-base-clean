1.Our goals this week

Week 1 gave Dan — the retina surgeon this project is built around — an assistant that reads his medical-records system and answers questions about a patient, with every claim linked back to the chart. That worked, but only for information already typed into the system as structured data.

Real clinics don't run on tidy data. The scenario this week's assignment starts from: the physician is prepping for a follow-up visit. The chart looks unchanged — but the important news is buried in a scanned lab PDF and a paper intake form the front desk uploaded. The physician's question is simple: what changed, what should I pay attention to, and what evidence supports the recommendation?

Week 2 makes the assistant able to answer that, with two new abilities:

  • It can read paper. A scanned outside lab report or an intake form can now be uploaded, read, and turned into usable, checkable facts.
  • It can consult the practice's own rulebook. Questions like "what do our protocols say about this drug at this kidney function?" are answered from the practice's written protocols — with the exact passage quoted.

And because an assistant that quietly gets worse is dangerous, the third deliverable is proof: an automated test gate that blocks any code change that would degrade quality or safety. The governing rule is unchanged from Week 1: the system may be unavailable; it may never be silently wrong.

2.Our architectural approach

The medical-records system itself (OpenEMR — the practice's permanent system of record) is untouched. Everything new lives in a small separate service running beside it, called a sidecar — think of an annex built next to the records room, connected by a door. If the annex has a bad day, the records room is unaffected.

Inside the sidecar, this week's work is organized like a small team rather than one giant brain:

  • A supervisor looks at each request and makes one routing decision: plain chart question, "check the protocols" question, or "read this document" job.
  • Two specialist workers do the work: the intake-extractor reads uploaded documents; the evidence-retriever finds the relevant protocol passages.
  • A critic — an automated fact-checker — re-verifies every citation before anything reaches the user.

Every handoff between them is logged under a shared tracking number, so a reviewer can reconstruct exactly who did what for any single answer.

Why this shape? In a clinical setting, small and inspectable beats big and clever: one do-everything AI is impossible to audit, while three narrow roles with logged handoffs can each be tested and watched. The assignment itself says the best versions "feel narrower than the original spec and stronger because of it."

3.The data structures

Five kinds of information, each with a deliberate home:

The source document
The actual uploaded PDF or image, filed into the medical-records system itself — the permanent, authoritative artifact, exactly as received. The assistant keeps pointers to it, never a substitute for it.
Extracted facts
What the AI read out of the document — each lab value, medication, or allergy stored as a typed, structured value (a number is a number, a date is a date), each carrying a citation back to the exact spot on the exact page it came from.
The guideline corpus
Eight short authored documents in the project repository — seven clinical practice protocols (screening schedules, dosing thresholds, treatment rules, each citing the named real-world guideline it is grounded in) plus the practice's intake-documentation standard. Ordinary text files anyone can read, versioned like code.
The vector index
Each protocol passage gets an embedding — a numerical fingerprint of what the passage means, so the system can find passages by meaning rather than by matching words. New this week: these fingerprints are stored permanently in the database, so a restart reuses all 71 of them instead of recomputing anything.
The answer records
Every pre-visit brief and every chat answer is stored in full, with its citations, so any answer can be audited later against its sources.

The key principle: nothing is ever marked "true." Everything is marked "supported by this document, at this location." The system stores evidence and pointers, not beliefs.

4.What happens when a document is uploaded

When the front desk uploads, say, an outside kidney-function panel, it moves through a fixed series of checkpoints:

The upload pipeline

1Received
Uploading is a chart write, so it requires a signed-in, accountable clinical role.
▼
2Stored in the medical record
The original file goes into OpenEMR first — before any AI touches it.
▼
3Read into a strict form
A vision-capable AI reads the scan; its output must fit a rigid, typed form.
Doesn't fit the form (after one corrected retry)? The upload is marked failed and nothing is saved.
▼
4Every value located on the page
Non-AI code pins each value to its exact box, or its page — or honestly flags it "unverified."
Unverified values stay visible but can never be cited in an answer.
▼
5Patient check
The patient printed on the document is compared to the chart it's being attached to.
Mismatch? Everything is blocked.
▼
6Facts saved
Typed values persist, each with its citation and a confidence score.
▼
7Safety flags update
The new kidney number automatically re-tiers the medication-safety flag.

Step 7 is the payoff in our demo case: the extracted eGFR (a standard measure of kidney function) automatically re-tiers the patient's safety flag for hydroxychloroquine — a long-term medication whose risk of damaging the retina rises as kidney function falls — through the same rule-based safety engine built in Week 1. The paper changed the clinical answer, and the lab value behind it is cited.

One boundary worth stating: information that already lives in the records system as structured data skips all of this. It still takes the Week 1 path — read directly, no document processing.

5.How information goes into the "library"

The protocol side works like building a small reference library. (The technique's jargon name is RAG — retrieval-augmented generation — which just means "look it up before you answer.")

Each protocol document is split into passages, carefully, so every rule stays attached to its conditions: a dose threshold is never separated from the population it applies to, and each passage keeps its section heading. Split a rulebook carelessly and "reduce the dose" drifts away from "…in patients with kidney impairment" — exactly the error a clinic can't afford.

Each passage then gets its meaning-fingerprint (embedding) computed once and stored in the database. If a protocol is later edited, only the changed passages are re-processed; everything else is reused, and the whole index can be rebuilt from the repository's text files with one command.

One deliberate absence: patient documents never go into this library. Meaning-similarity answers "which passage is about this topic?" — a fine question for protocols, a dangerous one for patient data, where the bar is the exact value from the right lab report, with a citation. A "similar" lab row is a safety bug, not a convenience. Patient facts stay in their own structured store and are always retrieved exactly.

6.How it's queried

When a chat question sounds like a protocol question ("what screening interval do our protocols recommend…?"), the supervisor hands it to the evidence-retriever, which does four things:

  1. Scrubs the question. Anything identifying — names, dates of birth, record numbers — is stripped by plain rule-following code (not an AI) before the question leaves the system. Only de-identified clinical concepts (drug, dose range, disease stage) go out.
  2. Searches two ways at once. An exact-keyword search (strong on drug names and precise terms) runs alongside a meaning-similarity search over the fingerprints (strong when the question is phrased differently from the document). Clinical language needs both — the same drug appears as "Plaquenil" and "hydroxychloroquine."
  3. Merges the two result lists with a standard rank-combining formula.
  4. Re-ranks with a specialist. A dedicated re-ranking service (from Cohere, an AI company whose model does exactly this one job) reads the merged candidates against the question and puts the genuinely relevant passages on top.

Only the top few passages — five at most — reach the answering AI. And the whole protocol-checking turn runs inside a strict time budget: if the library is unreachable, the assistant says so and answers from the record only, rather than stalling or pretending.

7.How answers come out

The answering AI works under open-book-exam rules: it may use only the passages it was handed, and every claim must quote its source word-for-word, with machine-readable pointers to the document, section, and passage.

Before the user sees anything, the critic replays the exam. It compares each quoted snippet — character by character — against the stored original. A claim whose quote doesn't match, or that carries no citation at all, is withheld on the server; it never renders.

And when the library simply has nothing on a topic, the honest answer ships: "No practice protocol on file covers this question." The assistant never falls back on its general training to invent a plausible-sounding protocol — an invented protocol delivered confidently would be the worst failure a tool like this can have.

8.The controls on the output

No single safeguard is trusted to be perfect, so they are layered:

  • Strict forms. Nothing malformed is ever saved — extraction output fits the pre-defined form or is discarded entirely.
  • Location grounding. Every extracted value is pinned to its spot in the document or visibly marked unverified — and unverified is unusable in answers.
  • The patient-mismatch block. A document whose printed patient doesn't match the chart goes nowhere.
  • The critic's quote check. Every citation re-verified word-for-word against the stored original before display.
  • Honesty by default. "No protocol on file" when the library is empty on a topic; a refusal — offering the protocol-grounded alternative — when asked to skip the safety checks ("just tell me a dose to prescribe").
  • The automated test gate. 58 recorded test cases across six pass/fail categories run automatically on every proposed code change. Safety categories (refusing unsafe requests, citations present, no patient data in logs) block a change on even one newly failing case; quality categories block on any meaningful drop. We rehearsed the gate by deliberately injecting three faults — a broken form field, a dropped citation, a planted identifier in a log line. It caught all three.
  • A spending cap. AI usage is metered against a hard $5-per-day budget — the system stops spending rather than running up a bill.
  • Privacy rules with teeth. Logs and monitoring traces carry IDs and counts — never patient names, never document text, never clinical values. An automated check scans for leaks on every code change.

9.Where it stands

Everything described above is live on the deployed demo: upload a lab PDF and watch the checkpoints advance, click a value to see it highlighted on the scanned page, ask the protocol question, and read the cited answer. Every patient, chart, and document in the system is synthetic — no real patient information appears anywhere.
