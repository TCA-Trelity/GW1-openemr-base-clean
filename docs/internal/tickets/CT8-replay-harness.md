## Why (and why not yet)

Hand-written goldens encode what the authors *imagined* users ask; replayed
real questions catch the regressions imagination misses (phrasing the router
mis-tiers, ambiguity the composer over-answers). But today there is **no
traffic worth replaying**: the deployment is a single-surgeon demo whose
Langfuse traces are overwhelmingly the team's own smoke tests and grader
walkthroughs — replaying those would re-encode the authored goldens with
extra steps. The harness becomes valuable exactly when organic pilot traffic
exists; speccing it now is cheap, building it now is waste.

## Design constraints (the load-bearing part of this spec)

### 1. PHI scrubbing rules (G18 — non-negotiable)

`docs/w2/requirements.md` G18 (line 661): "traces, logs, **eval datasets**,
and cost reports contain no patient identifiers, no raw document text, no
extracted clinical values" — a replay case file IS an eval dataset and is
bound by it. Committed replay cases therefore carry **ids, hashes, and
counts** plus a scrubbed question string, never raw context:

- Reuse the existing scrubber as the first pass: `sidecar/src/retrieval/queryPolicy.ts:34 scrubQuery(raw: string, identifiers: PatientIdentifiers = {}): string` (strips the launch patient's name parts/DOB/MRN and identifier patterns; the `no_phi_in_logs` eval family already asserts a planted identifier cannot survive it).
- Second pass, mandatory because real questions can embed OTHER people's identifiers the patient-scoped scrubber cannot know ("my sister Ana Maria had this"): a **human review gate** — every sampled question is eyeballed and either approved, hand-redacted, or dropped before commit. No fully-automatic import path, ever.
- Provenance stays linkable without leaking: each case records `source_trace_hash` (SHA-256 of the trace id — reviewable by someone with Langfuse access, meaningless in the repo), sample window date, routing tier observed. Never the raw trace id in the committed file (trace ids are correlation ids that also appear in ops logs; a hash is enough).
- The committed file must pass the same canary discipline as everything else: extend the PHI sweep (`eval/phi-log-sweep.eval.ts` family) to sweep the replay case file for the planted canaries + date-of-birth/MRN patterns.

### 2. Sampling policy

- **Window:** on pickup, sample from the most recent N weeks of organic traffic only (exclude traces generated by CI smoke, `load-test.ts`, and team sessions — filterable by the dev-token username field and known test patient ids; the exclusion list is part of the harness).
- **Stratify by routing tier** so the set mirrors real mix instead of over-sampling the easy fast path: the router's three outcomes `fast_path | needs_evidence | needs_extraction` (supervisor-as-entry, locked #4; observed tier rides the trace's routing metadata).
- **Dedupe:** normalize (lowercase, strip whitespace/punctuation) then hash; one case per hash; cap per import (e.g. 15) so curation stays a human-scale task.
- **Selection preference** inside each stratum: questions whose traces showed degraded outcomes first (`run_success=0`, `citations_failed>0`, `unverified_count>0` — the CT6 error-analysis log is the natural feeder: its "new-eval-case" dispositions ARE replay candidates).

### 3. How replay cases join the difficulty scheme (CT2)

CT2 (prerequisite) makes `difficulty: 'straightforward' | 'ambiguous' | 'edge-case'` a required field on `EvalRecord` (`sidecar/eval/collector.ts:16` — today the interface has `category`/`enforce` but no difficulty; verified). Replay cases:

- Get difficulty assigned at **curation time by the human reviewer** (the same person doing the PHI review) — real questions default to `ambiguous` unless obviously otherwise, because organically-phrased questions are exactly the ambiguity tier hand-written cases under-represent. No auto-tagging heuristic pretending to judge difficulty.
- Carry `category` like every other case (mostly `retrieval_grounded` / `factually_consistent` for evidence questions; `safe_refusal` for out-of-scope asks that got refused — refusal replays are especially valuable).
- Appear in CT2's coverage-by-difficulty report section like any other case — no special-casing in the report.

### 4. What a replay case asserts (deterministic-gate compatible)

Replayed questions have no authored "expected answer", so cases assert
**structural invariants**, not verbatim text — keeping them eligible for the
deterministic gate (G17: offline retrieval + scripted/stubbed model seams,
exactly the graph-path.eval.ts pattern):

- routing tier matches the curated expectation (router regression catch);
- for evidence questions: retrieval over the committed corpus returns the curated doc/chunk (hit) — reusing the retrieval-goldens comparator;
- for refusal questions: the refusal/deflection behavior holds;
- citation-gate outcomes (no unverified citation released) hold on the replayed path.
Live answer-quality drift on replayed questions is CT7's judge territory (informational), not the gate's.

### 5. Shape (when built)

- `sidecar/eval/replay/replay-cases.json` — the curated, scrubbed case data (id, scrubbed question, source_trace_hash, sampled_at, routing_expected, category, difficulty, assertion params).
- `sidecar/eval/replay-grounded.eval.ts` — one suite iterating the file through the structural assertions, recording via `recordEval` with `enforce: 'soft'` for the first import cycle (quality-tier baseline math applies), hardening later per category rules.
- `sidecar/src/scripts/sample-replay-candidates.ts` — the sampling script (reads Langfuse public API read-only like CT6's `fetch-recent-traces.ts`, applies §2, emits a REVIEW file — never writes into `eval/` itself; the human moves approved cases in).
- Baseline impact: adding cases changes category totals ⇒ the import lands with its own `npm run eval:baseline` re-baseline commit, separate and labeled (CT5 rules).

## What NOT to do

- Do NOT build any of this before the pickup trigger fires — the merged plan explicitly keeps it unscheduled; half-building it now creates maintenance surface with zero replay value.
- Do NOT ever add an automatic trace→eval import path (no cron, no CI sampling) — the human PHI-review gate (§1) is structural, not a phase-one shortcut.
- Do NOT commit raw trace ids, raw questions pre-review, or any answer text from traces into the repo (G18; eval datasets are in scope of the privacy audit).
- Do NOT assert verbatim answers in replay cases (§4) — that would either couple the gate to live model output (flaky, forbidden) or require authoring answers, which defeats "replay".
- Do NOT count replayed team/CI traffic as "real traffic" when evaluating the pickup trigger.
